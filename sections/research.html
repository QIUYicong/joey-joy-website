本站数据架构 · 方案对比
2026年5月18日
Featured
目前网站的内容（作品、研究、随笔、相册）都写死在 HTML 文件里。要让内容可增删改、跨页面复用、未来支持搜索/分页，需要把数据层抽出来。下面对比四个由轻到重的方案，每个都给出适用场景与成本。
方案 A · 静态 JSON
Easiest
怎么做：把所有内容写进 data/articles.json，前端用 fetch() 读取并渲染。部署到 GitHub Pages / Vercel / Netlify。
✓ 零成本零运维 · 改内容就是改 JSON 推一下 Git · 速度极快 · 适合个人小站
✗ 需要会用 Git · 没有后台编辑器 · 不适合多人协作
方案 B · Notion 作 CMS
Friendly
怎么做：把内容写在 Notion 数据库里，通过官方 API（或 notion-sdk-js）拉取。可加缓存层降低延迟。
✓ 编辑体验最好 · 富文本所见即所得 · 手机也能改 · 免费
✗ 需要 Notion token · API 限速 3 次/秒 · 首屏慢，建议加 ISR/缓存
方案 C · Supabase
Recommended
怎么做：用 Supabase 托管的 Postgres，自动生成 REST/GraphQL 接口。带认证、图片存储、可视化后台。
✓ 真正的关系数据库 · 免费层够个人站 · 有管理后台 · 支持评论、点赞等功能
✗ 要学一点 SQL · 配置 RLS 安全策略需谨慎
方案 D · 自建后端
Full Control
怎么做：Node.js (Express / Hono) + MongoDB 或 Postgres，部署在 Railway / Render / Fly.io。
✓ 完全自由，想加什么逻辑都行 · 适合长期演化的项目
✗ 维护成本最高 · 数据库账户、备份、监控都要自己管 · 起步最慢
建议路径：先用 方案 A（静态 JSON）把现有内容迁出来 — 半天工作量，立即解决「改内容要改 HTML」的痛点。等内容量超过 30 篇、或需要评论/搜索功能时，再无痛切到 方案 C（Supabase）— 因为 JSON 与 Supabase 的字段结构可以一一对应，迁移只是改数据源。
